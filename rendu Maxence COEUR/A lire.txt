1/ Méthodes HTTP : PUT vs PATCH
PUT (Remplacement) : Remplace la ressource cible dans sa totalité. L'objet envoyé écrase l'ancien. Si un champ est omis, il peut être supprimé ou réinitialisé par le serveur.
PATCH (Modification) : Applique une modification partielle. On ne transmet que les données à mettre à jour (ex: seulement le champ prix ou stock).

2/ Sécurité & Navigateur : Le CORS
Définition : Le Cross-Origin Resource Sharing est une sécurité appliquée par le navigateur. Elle empêche un script provenant d'un domaine A d'accéder aux ressources d'un domaine B sans permission explicite.
Comportement : Firefox (ou Chrome) bloque la requête si l'API ne renvoie pas les headers Access-Control-Allow-Origin autorisant le front-end.
Outils tiers : Postman ou Insomnia ne sont pas des navigateurs ; ils ignorent donc ces restrictions de sécurité, ce qui explique pourquoi la requête peut fonctionner sur Postman mais échouer sur un site web.

3/ Architecture : Reverse Proxy (Nginx/Apache)
Placer un serveur comme Nginx devant une application Node.js répond à trois besoins :
Sécurité : Masquage de l'IP du serveur applicatif et gestion centralisée du SSL/TLS (HTTPS).
Performance : Prise en charge des fichiers statiques (CSS, images, JS) pour décharger Node.js.
Évolutivité (Load Balancing) : Répartition de la charge entre plusieurs instances du serveur pour éviter la saturation.

4/ Optimisation des Performances
Pagination : Utilisation de limit et skip (ou pagination par curseur) pour ne traiter que de petits segments de données à la fois.
Indexation (MongoDB) : Création d'index sur les champs fréquemment consultés (ex: email, id_utilisateur) pour éviter le scan intégral de la collection.
Mise en cache (Caching) : Utilisation de Redis (stockage en RAM) ou de headers Cache-Control pour servir instantanément des données qui changent peu, sans solliciter la base de données.
